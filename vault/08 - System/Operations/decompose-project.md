# Operation: decompose-project

> Turn an Overview into 4ÔÇô8 phase files. Either fresh decomposition (`onyx plan <project>`) or extension (`onyx plan <project> --extend`). Shared phase-decompose skill handles the generation; this operation wires it to the bundle.

## Preconditions
- Project bundle exists with `Overview.md` at root.
- Overview has a non-empty goals section (`## Goals`, `## Scope`, or `## Description`).
- Overview frontmatter has `profile:` set (required to pick code-vs-generic template).
- For engineering/trading profiles: `repo_path:` set and existing on disk (or resolvable via `reposRoot` fuzzy match).

## Invocation context
- Explicit only: `onyx plan <project>` (whole-project, fresh) or `onyx plan <project> --extend` (append to existing plan).
- Not dispatched by [[08 - System/Operations/route.md|route]] ÔÇö this is project-scoped, not phase-scoped.

## Read order
1. `<bundle>/Overview.md` ÔÇö frontmatter + body (Goals, Constraints, Success Criteria).
2. `08 - System/Profiles/<profile>.md` ÔÇö template selection + required fields.
3. `<bundle>/Knowledge.md` ÔÇö prior learnings that inform phase structure.
4. `<bundle>/Phases/*.md` ÔÇö existing phases (for `--extend` mode; need current max phase number).
5. For code profiles: repo tree via [[08 - System/Agent Skills/_onyx-runtime/repo-scan/SKILL.md|repo-scan]].
6. For non-code profiles: `<bundle>/Source Context.md` and other reference docs if present.

## Procedure

### Step 1 ÔÇö Load Overview + profile
Read Overview frontmatter. Extract `profile`, `repo_path` (code), any `priority`/`constraints`/`success_criteria`. Mode = `fresh` if Overview has no `Phases/*.md` children OR if `--fresh` flag passed; else `extend`.

### Step 2 ÔÇö Resolve repo path (code profiles)
Same algorithm as [[08 - System/Operations/atomise.md|atomise]]:
1. Overview frontmatter `repo_path` if it exists on disk.
2. Fuzzy match against `reposRoot` children.
3. Fail fast if unresolvable Ôćĺ return error; caller should block.

### Step 3 ÔÇö Determine starting phase number
- `fresh` mode: start at 1.
- `extend` mode: scan `<bundle>/Phases/`. Find max `phase_number` across all files (parse from frontmatter; fall back to filename `P<N>`). Start at `max + 1`.

### Step 4 ÔÇö Generate phase stubs
Invoke [[08 - System/Agent Skills/_onyx-runtime/phase-decompose/SKILL.md|phase-decompose]] with:
- `project_id`, `profile_name`, `is_code_profile`
- `repo_path` (code) or `bundle_path` (non-code)
- `overview_content` (full body)
- `knowledge_content` (Knowledge.md body, capped at 2KB)
- `existing_phases_summary` (for extend mode ÔÇö list of existing phase names + numbers)
- `source_context` (repo tree OR bundle listing)
- `start_number` (from Step 3)
- `mode` = `"fresh"` | `"extend"`

Skill returns: list of phase specs (each with `number`, `name`, `summary`, `context`, `acceptance_criteria`), OR list of written phase file paths if it used the direct-agent-write branch.

### Step 6 ÔÇö Update Overview
If `fresh` mode and Overview `status:` is `draft` Ôćĺ transition to `active`. Bump `updated:`.

If `extend` mode Ôćĺ just bump `updated:`.

Add a note to Overview body (if not present) linking to the Phases Hub:
```
See [[<project_id> - Phases Hub|Phases Hub]] for the full phase list.
```

### Step 7 ÔÇö Auto-atomise trivial phases (optional)
For each newly created phase with `depends_on: []` (i.e. P1 in fresh mode, or any leaf): offer to invoke [[08 - System/Operations/atomise.md|atomise]] inline. Default: off ÔÇö atomise is a separate explicit step. Override with `--atomise-first` flag.

### Step 8 ÔÇö Log
Call `tools/write-exec-log.sh`:
```
--status PLAN
--project <project_id>
--phase -
--summary "phases=<count> mode=<fresh|extend> profile=<name>"
```

## Post-conditions & transitions
- 3+ new phase files created under `<bundle>/Phases/` (and matching log stubs under `<bundle>/Logs/`).
- Every new phase has valid frontmatter including `phase_number`, `phase_name`, `status: backlog`, `up: <project_id> - Phases Hub`.
- Overview `status:` transitioned to `active` if it was `draft` (fresh mode only).
- Phases Hub body updated to list new phases (heal-fractal-links covers this on next pass if not inlined here).

## Error handling
- **RECOVERABLE:** LLM timeout in Step 4 ÔÇö retry once.
- **BLOCKING:** Overview goals too vague to decompose Ôćĺ phase-decompose returns `ambiguous_scope` Ôćĺ write `## Human Requirements` on Overview, exit.
- **INTEGRITY:** repo_path unresolvable for code profile; Overview frontmatter corrupt.

- [[08 - System/Agent Skills/_onyx-runtime/repo-scan/SKILL.md|repo-scan]] ÔÇö for code profiles in Step 1.

## Tools invoked

## Native primitives relied on
- **Read** ÔÇö Overview, Profile, Knowledge, existing phases.
- **Write** ÔÇö new phase files, log stubs.
- **Edit** ÔÇö Overview status transition + updated:.
- **Bash** ÔÇö optional `git log` for engineering repo context (via allowed_shell).

## Acceptance (self-check before exit)
- `<bundle>/Phases/` contains N new files with valid frontmatter.
- Every file's `phase_number` is unique.
- Every file's `depends_on` references an existing phase (or is empty for P1).
- Every Overview acceptance criterion is covered by at least one phase.

## ExecLog entry format
```
<ISO> | <project> | - | PLAN | <duration-sec> | phases=<count> mode=<mode> profile=<name>
```

## Shadow-mode comparison criteria
For same Overview + repo state:
- Phase count within ┬▒1.
- Every phase name in TS output has a semantic-equivalent in directive output (acceptable: synonym or reorder of verb+object).
- Acceptance-criteria coverage of Overview goals matches.
- Order preserves dependency chain (P<k> only depends on phases < k).
